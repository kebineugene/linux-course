📖 Логи — память системы

Зачем админу логи?

Всё, что происходит в системе, пишется в логи. Умение их читать и анализировать — 50% работы админа. Без логов вы слепы: не знаете, кто заходил, что падало, где ошибка.

  • 🔍 Диагностика — почему сервис упал? кто атакует сервер?
  • 🛡️ Безопасность — кто пытался войти? какие команды выполнял?
  • 📊 Аудит — что изменилось в системе? когда?
  • ⚡ Производительность — какие запросы медленные? где узкое место?
💡 Аналогия: Логи — это чёрный ящик самолёта. Когда всё работает — их никто не читает. Когда что-то падает — это единственный способ понять, что произошло.

️ syslog и journald

Традиционный syslog

Старая система логирования. Логи хранятся в текстовых файлах в /var/log/. Простая, понятная, работает везде.

tail -f /var/log/syslog # следить за системным логом в реальном времени less /var/log/auth.log # логи аутентификации (входы, sudo) less /var/log/kern.log # сообщения ядра less /var/log/dpkg.log # установка/удаление пакетов less /var/log/nginx/access.log # запросы к веб-серверу less /var/log/nginx/error.log # ошибки веб-сервера
🧠 Основные файлы логов:
  • /var/log/syslog — общий системный лог (Ubuntu/Debian)
  • /var/log/messages — общий системный лог (CentOS/RHEL)
  • /var/log/auth.log — аутентификация, SSH, sudo
  • /var/log/kern.log — ядро, драйверы, hardware
  • /var/log/dpkg.log — пакетный менеджер

journalctl (systemd)

Современная система логирования. Логи хранятся в бинарном формате, но journalctl делает их удобными для чтения. Преимущества: централизованный сбор, быстрый поиск, структурированные данные.

sudo journalctl -u nginx # логи сервиса nginx sudo journalctl -u nginx --since "10 min ago" # за последние 10 минут sudo journalctl -u nginx --since today # за сегодня sudo journalctl -u nginx --since "2024-01-15 14:30:00" # с конкретного времени sudo journalctl -f # следить за всеми логами в реальном времени sudo journalctl -p err # только ошибки sudo journalctl -p warning # предупреждения и ошибки sudo journalctl -n 50 # последние 50 строк sudo journalctl --no-pager # без постраничного вывода (для скриптов)
💡 journalctl vs syslog:
  • journalctl — быстрее, удобнее поиск, есть метаданные (PID, UID, приоритет)
  • syslog — текстовые файлы, можно читать любыми инструментами (grep, awk)
  • На современных системах работают обе — выбирайте удобную

🔬 Практика: просмотр логов

Пошаговая практика

# 1. Системный лог tail -20 /var/log/syslog # 2. Логи сервиса sudo journalctl -u ssh --since "1 hour ago" # 3. Живой просмотр (следить за логами) sudo journalctl -f # 4. Только ошибки sudo journalctl -p err --since today # 5. Логи аутентификации (кто заходил?) sudo tail -50 /var/log/auth.log

Уровни логирования

Зачем нужны уровни?

Не все сообщения одинаково важны. Уровни помогают фильтровать шум и находить критичные проблемы.

Уровень          Приоритет  Когда использовать
─────────────────────────────────────────────────
DEBUG            7          Отладка, детали работы
INFO             6          Обычные события (запуск, остановка)
NOTICE           5          Нормальные, но важные события
WARNING          4          Потенциальные проблемы
ERROR            3          Ошибки (что-то не работает)
CRITICAL         2          Критичные ошибки (сервис падает)
ALERT            1          Требует немедленного действия
EMERGENCY        0          Система не работает

Как менять уровень логирования?

В systemd-сервисах можно задать уровень через переменную окружения:

# В unit-файле /etc/systemd/system/myapp.service: [Service] Environment=LOG_LEVEL=debug # После изменения: sudo systemctl daemon-reload sudo systemctl restart myapp

В nginx уровень задаётся в конфиге:

# В /etc/nginx/nginx.conf: error_log /var/log/nginx/error.log warn; # только warning и выше # Варианты: debug, info, notice, warn, error, crit, alert, emerg
💡 Совет: На production ставьте уровень warn или error. Уровень debug создаёт огромные логи и замедляет работу.

🛠️ Инструменты для анализа логов

awk — извлечение столбцов

awk разбивает строку на поля (по пробелам по умолчанию) и позволяет работать с каждым полем отдельно.

awk '{print $1}' access.log # первое поле (обычно IP) awk '{print $1, $3}' access.log # первое и третье поля awk '$3 > 1000 {print $1}' access.log # если третье поле > 1000, вывести первое awk -F: '{print $1}' /etc/passwd # разделитель — двоеточие

sed — замена в потоке

sed 's/error/ERROR/g' log.txt # заменить "error" на "ERROR" sed -n '/error/p' log.txt # вывести только строки с "error" sed '/error/d' log.txt # удалить строки с "error"

cut, sort, uniq — комбинаторика

# Топ IP-адресов из access.log: cut -d' ' -f1 access.log | sort | uniq -c | sort -rn | head -10 # Расшифровка: # cut -d' ' -f1 — вырезать первое поле (разделитель — пробел) # sort — отсортировать (для uniq нужно) # uniq -c — подсчитать повторы # sort -rn — отсортировать по числу (обратно) # head -10 — топ-10
🧠 Мнемоника: cut → sort → uniq → sort → head — стандартный конвейер для подсчёта повторов.

grep — поиск паттернов

grep "error" /var/log/syslog # найти "error" grep -i "error" log.txt # без учёта регистра grep -v "error" log.txt # строки БЕЗ "error" grep -n "error" log.txt # с номерами строк grep -r "root" /etc/ # во всех файлах рекурсивно grep -c "error" log.txt # подсчитать количество строк grep -E "error|warning" log.txt # регулярное выражение (error ИЛИ warning)

🔬 Практика: парсинг логов

Пошаговая практика

# 1. Топ-5 IP из access.log cut -d' ' -f1 /var/log/nginx/access.log | sort | uniq -c | sort -rn | head -5 # 2. Все ошибки за сегодня sudo journalctl -p err --since today # 3. Количество ошибок в syslog grep -c "error" /var/log/syslog # 4. Строки с "Failed password" (попытки взлома SSH) grep "Failed password" /var/log/auth.log | tail -20 # 5. Уникальные IP, пытавшиеся войти по SSH grep "Failed password" /var/log/auth.log | awk '{print $11}' | sort -u

📦 JSON-логи и jq

Зачем JSON вместо текста?

Современные приложения часто пишут логи в формате JSON. Это структурированные данные, которые легко парсить машиной.

# Текстовый лог (сложно парсить): 2024-01-15 14:30:15 ERROR user=alex action=login status=failed # JSON-лог (легко парсить): {"timestamp":"2024-01-15T14:30:15","level":"ERROR","user":"alex","action":"login","status":"failed"}

jq — парсер JSON

sudo apt install -y jq # Красивый вывод JSON: cat log.json | jq . # Извлечь поле: cat log.json | jq '.user' # Извлечь несколько полей: cat log.json | jq '{user, action}' # Фильтрация: cat log.json | jq 'select(.level == "ERROR")' # Подсчёт: cat log.json | jq -s 'group_by(.user) | map({user: .[0].user, count: length})'
💡 Совет: Если логи в JSON — используйте jq, а не awk/grep. Это надёжнее и проще.

🔄 Ротация логов

Зачем нужна ротация?

Логи растут бесконечно. Без ротации диск заполнится. logrotate автоматически сжимает и удаляет старые логи.

cat /etc/logrotate.conf # главный конфиг ls /etc/logrotate.d/ # конфиги для отдельных сервисов

Пример конфига logrotate

# Файл: /etc/logrotate.d/myapp /var/log/myapp/*.log { daily # ротировать каждый день missingok # не ошибаться, если файл отсутствует rotate 7 # хранить 7 старых файлов compress # сжимать старые логи (gzip) delaycompress # сжимать только со второго раза notifempty # не ротировать пустые файлы create 0640 www-data adm # права на новый файл sharedscripts # скрипты выполняются один раз postrotate systemctl reload nginx > /dev/null 2>&1 || true endscript }
🧠 Параметры:
  • daily/weekly/monthly — частота ротации
  • rotate N — сколько старых файлов хранить
  • compress — сжимать через gzip
  • size 100M — ротировать, если файл больше 100МБ
  • postrotate — команда после ротации (например, перезапустить сервис)

Тестирование logrotate

sudo logrotate -d /etc/logrotate.conf # сухой прогон (без изменений) sudo logrotate -f /etc/logrotate.d/myapp # принудительная ротация

🎯 Реальные сценарии

Сценарий 1: «Сайт упал» — диагностика через логи

# Шаг 1: Проверить статус сервиса sudo systemctl status nginx # Шаг 2: Посмотреть логи сервиса за последние 10 минут sudo journalctl -u nginx --since "10 min ago" # Шаг 3: Посмотреть ошибки nginx sudo tail -50 /var/log/nginx/error.log # Шаг 4: Найти конкретную ошибку sudo grep "error" /var/log/nginx/error.log | tail -20 # Шаг 5: Посмотреть access.log — какие запросы приходят? sudo tail -20 /var/log/nginx/access.log # Шаг 6: Найти 500-е ошибки sudo awk '$9 == 500' /var/log/nginx/access.log | tail -20

Сценарий 2: «Кто-то пытается взломать сервер»

# 1. Посмотреть неудачные попытки входа по SSH sudo grep "Failed password" /var/log/auth.log | tail -30 # 2. Найти топ IP, пытавшихся войти sudo grep "Failed password" /var/log/auth.log | awk '{print $11}' | sort | uniq -c | sort -rn | head -10 # 3. Посмотреть успешные входы sudo grep "Accepted" /var/log/auth.log | tail -20 # 4. Проверить, кто сейчас залогинен who last -20
💡 Защита: Если видите много попыток входа — установите fail2ban (модуль 6).

Сценарий 3: «Диск заполнился» — найти виновника

# 1. Проверить, какой раздел заполнен df -h # 2. Найти большие логи sudo find /var/log -type f -size +100M -exec ls -lh {} \; # 3. Посмотреть размер логов по папкам sudo du -sh /var/log/* # 4. Проверить, не забыли ли ротацию ls -lh /var/log/nginx/ # 5. Очистить старые логи (осторожно!) sudo find /var/log -name "*.gz" -mtime +30 -delete

🔧 Troubleshooting: Если не получилось

❓ "journalctl показывает мало строк"
  • По умолчанию journalctl хранит логи только с последней загрузки
  • Включите постоянное хранение: sudo mkdir -p /var/log/journal
  • Перезапустите: sudo systemctl restart systemd-journald
  • Или настройте в /etc/systemd/journald.conf: Storage=persistent
❓ "Логи не пишутся"
  • Проверьте место на диске: df -h
  • Проверьте inodes: df -i
  • Проверьте права на файл лога: ls -l /var/log/...
  • Проверьте, запущен ли сервис логирования: systemctl status rsyslog или systemctl status systemd-journald
❓ "Как найти ошибку за конкретное время?"
  • Используйте --since и --until:
    sudo journalctl -u nginx --since "2024-01-15 14:00:00" --until "2024-01-15 15:00:00"
  • Или через grep с контекстом:
    grep "14:30" /var/log/syslog | grep "error"
❓ "Логи слишком большие, диск заполняется"
  • Настройте logrotate (см. раздел выше)
  • Для journalctl: sudo journalctl --vacuum-size=100M — оставить только 100МБ
  • Или в конфиге: SystemMaxUse=100M в /etc/systemd/journald.conf
  • Найдите большие логи: sudo find /var/log -type f -size +100M
❓ "logrotate не работает"
  • Проверьте синтаксис: sudo logrotate -d /etc/logrotate.conf (сухой прогон)
  • Проверьте, что cron запускает logrotate: ls /etc/cron.daily/logrotate
  • Запустите вручную: sudo logrotate -f /etc/logrotate.d/myapp
  • Проверьте права на конфиг: должен быть root:root с правами 644
Универсальный совет: Если не можете найти ошибку в логах — увеличьте уровень логирования до debug, воспроизведите проблему, потом верните уровень обратно. В debug-режиме видно всё.

Базовые задания
