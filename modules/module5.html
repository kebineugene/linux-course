👻 Что такое демон?

Зачем админу systemd?

Демон — фоновый процесс, который работает без интерактивного ввода. systemd — современная система инициализации, которая управляет демонами, сокетами, таймерами. Это сердце современного Linux.

  • Автозапуск — сервисы запускаются при загрузке системы
  • 🔄 Автоматический рестарт — если сервис упал, systemd его поднимет
  • 📊 Логи — централизованный сбор логов всех сервисов
  • ⏰ Таймеры — замена cron для регулярных задач
💡 Аналогия: systemd — это диспетчер в аэропорту. Он знает, какие рейсы (сервисы) должны лететь, когда их запускать, что делать при задержках, и ведёт журнал всех событий.

📖 Теория: systemd

Unit-файлы: анатомия сервиса

Каждый сервис описывается unit-файлом в /etc/systemd/system/ или /lib/systemd/system/. Структура:

[Unit] Description=My Application After=network.target Wants=network.target [Service] Type=simple User=www-data WorkingDirectory=/var/www/myapp ExecStart=/usr/bin/python3 app.py ExecReload=/bin/kill -HUP $MAINPID Restart=always RestartSec=5 StandardOutput=journal StandardError=journal [Install] WantedBy=multi-user.target
🧠 Разбор секций:
  • [Unit] — метаинформация, зависимости
  • [Service] — как запускать, от кого, что делать при падении
  • [Install] — когда запускать (при какой цели системы)

Типы сервисов (Type)

Type=simple # стандартный (процесс не форкается) Type=forking # процесс создаёт дочерний и выходит (старые демоны) Type=oneshot # выполнить один раз и выйти (скрипты инициализации) Type=notify # сервис сам сообщает о готовности

Цели системы (Targets)

Targets — это "состояния" системы, аналог runlevels в old init:

multi-user.target # многопользовательский режим без GUI (серверы) graphical.target # с графическим интерфейсом rescue.target # режим восстановления emergency.target # аварийный режим (только root)
💡 Для серверов обычно используется multi-user.target. GUI на серверах не нужен — экономит ресурсы.

Политика рестарта

Restart=no # не перезапускать (по умолчанию) Restart=always # всегда перезапускать Restart=on-failure # только при ошибке (non-zero exit code) RestartSec=5 # ждать 5 секунд перед рестартом StartLimitBurst=5 # максимум 5 попыток за период StartLimitIntervalSec=60 # период для подсчёта попыток
⚠️ Важно: Если сервис падает сразу после запуска, systemd может бесконечно его перезапускать. Используйте StartLimitBurst для ограничения.

🔬 Практика: unit-файлы

Создание своего сервиса

# 1. Создаём скрипт sudo nano /usr/local/bin/hello-service.sh #!/bin/bash while true; do echo "$(date): Hello from service" >> /tmp/hello.log sleep 10 done # 2. Делаем исполняемым sudo chmod +x /usr/local/bin/hello-service.sh # 3. Создаём unit-файл sudo nano /etc/systemd/system/hello.service [Unit] Description=Hello Service After=network.target [Service] ExecStart=/usr/local/bin/hello-service.sh Restart=always RestartSec=5 [Install] WantedBy=multi-user.target # 4. Активируем сервис sudo systemctl daemon-reload sudo systemctl start hello sudo systemctl enable hello sudo systemctl status hello # 5. Проверяем логи tail -f /tmp/hello.log

Управление сервисом

sudo systemctl status hello # статус sudo systemctl stop hello # остановить sudo systemctl start hello # запустить sudo systemctl restart hello # перезапустить sudo systemctl reload hello # перечитать конфиги (если поддерживает) sudo systemctl disable hello # убрать из автозагрузки sudo systemctl mask hello # полностью заблокировать sudo systemctl unmask hello # разблокировать

📋 Логи: journalctl

Централизованные логи systemd

systemd собирает логи всех сервисов в единый журнал. Это мощнее, чем старые текстовые логи в /var/log/.

sudo journalctl -u hello # логи сервиса hello sudo journalctl -u hello -f # следить в реальном времени sudo journalctl -u hello --since today # логи за сегодня sudo journalctl -u hello --since "1 hour ago" # за последний час sudo journalctl -u hello -n 50 # последние 50 строк sudo journalctl -u hello -p err # только ошибки sudo journalctl -u hello --no-pager # без постраничного вывода
💡 Критично для админа: Если сервис не запускается — первым делом смотрите journalctl -u <сервис> -n 50. Там будет причина ошибки.

Практика с логами

# Посмотреть все логи сервиса sudo journalctl -u hello # Следить за логами в реальном времени (как tail -f) sudo journalctl -u hello -f # Найти ошибки sudo journalctl -u hello -p err # Логи за последний час sudo journalctl -u hello --since "1 hour ago" # Экспорт логов в файл sudo journalctl -u hello > /tmp/hello-logs.txt

Теория: Таймеры

systemd timer vs cron

systemd таймеры — современная замена cron. Преимущества:

  • 📊 Логи — автоматически пишутся в journalctl
  • 🔄 Зависимости — можно указать, что таймер зависит от сервиса
  • ️ Точность — можно задать случайную задержку (RandomizedDelaySec)
  • Мониторинг — systemctl list-timers показывает все таймеры

Создание таймера

Таймер состоит из двух файлов: .service (что делать) и .timer (когда делать).

# Файл: /etc/systemd/system/backup.timer [Unit] Description=Daily Backup Timer [Timer] OnCalendar=daily RandomizedDelaySec=1h Persistent=true [Install] WantedBy=timers.target
🧠 Параметры таймера:
  • OnCalendar=daily — каждый день в полночь
  • OnCalendar=*:0/10 — каждые 10 минут
  • OnCalendar=Mon *-*-* 09:00:00 — каждый понедельник в 9:00
  • Persistent=true — выполнить пропущенные задания после простоя
  • RandomizedDelaySec=1h — случайная задержка до 1 часа (чтобы не все серверы одновременно запускались)

Управление таймерами

sudo systemctl daemon-reload sudo systemctl start backup.timer sudo systemctl enable backup.timer systemctl list-timers # показать все таймеры systemctl list-timers --all # включая неактивные sudo systemctl stop backup.timer sudo systemctl disable backup.timer

🔬 Практика: таймеры

Создание таймера для бэкапа

# 1. Создаём сервис sudo nano /etc/systemd/system/backup.service [Unit] Description=Backup Service [Service] Type=oneshot ExecStart=/bin/bash -c 'tar -czf /tmp/backup-$(date +%Y%m%d).tar.gz /home/user/documents' # 2. Создаём таймер sudo nano /etc/systemd/system/backup.timer [Unit] Description=Daily Backup Timer [Timer] OnCalendar=*-*-* 02:00:00 Persistent=true [Install] WantedBy=timers.target # 3. Активируем sudo systemctl daemon-reload sudo systemctl start backup.timer sudo systemctl enable backup.timer # 4. Проверяем systemctl list-timers

🎯 Реальные сценарии

Сценарий 1: Создание сервиса для Node.js приложения

# 1. Создаём unit-файл sudo nano /etc/systemd/system/myapp.service [Unit] Description=My Node.js App After=network.target [Service] Type=simple User=www-data WorkingDirectory=/var/www/myapp ExecStart=/usr/bin/node /var/www/myapp/app.js Restart=always RestartSec=10 StandardOutput=journal StandardError=journal Environment=NODE_ENV=production [Install] WantedBy=multi-user.target # 2. Активируем sudo systemctl daemon-reload sudo systemctl start myapp sudo systemctl enable myapp # 3. Проверяем sudo systemctl status myapp sudo journalctl -u myapp -f

Сценарий 2: Сервис не запускается — диагностика

# 1. Проверяем статус sudo systemctl status myapp # 2. Смотрим логи sudo journalctl -u myapp -n 50 # 3. Проверяем синтаксис unit-файла sudo systemd-analyze verify /etc/systemd/system/myapp.service # 4. Проверяем зависимости sudo systemctl list-dependencies myapp # 5. Тестовый запуск вручную sudo -u www-data /usr/bin/node /var/www/myapp/app.js

Сценарий 3: Настройка автоматического бэкапа

# Создаём скрипт бэкапа sudo nano /usr/local/bin/backup.sh #!/bin/bash DATE=$(date +%Y%m%d_%H%M%S) tar -czf /backup/db-$DATE.tar.gz /var/lib/mysql find /backup -mtime +7 -delete # удалить старые бэкапы # Делаем исполняемым sudo chmod +x /usr/local/bin/backup.sh # Создаём timer (как в практике выше) # Запускаем каждый день в 3:00

🔧 Troubleshooting: Если не получилось

❓ "Failed to start <service>.service"
  • Сервис не запустился
  • Смотрите логи: sudo journalctl -u <сервис> -n 50
  • Проверьте синтаксис: sudo systemd-analyze verify /etc/systemd/system/<сервис>.service
  • Проверьте права на исполняемый файл
❓ "Unit file not found"
  • Unit-файл не найден
  • Проверьте путь: /etc/systemd/system/ или /lib/systemd/system/
  • Выполните: sudo systemctl daemon-reload
  • Проверьте расширение файла (должно быть .service)
❓ Сервис постоянно перезапускается
  • Проверьте логи: sudo journalctl -u <сервис> -f
  • Возможно, ошибка в конфигурации приложения
  • Проверьте RestartSec — может быть слишком маленьким
  • Добавьте StartLimitBurst=3 для ограничения попыток
❓ "Permission denied" при запуске
  • Проверьте права на исполняемый файл: ls -l /path/to/script
  • Добавьте chmod +x
  • Проверьте пользователя в unit-файле (User=)
  • Проверьте SELinux/AppArmor, если включены
❓ Таймер не срабатывает
  • Проверьте, что таймер активен: systemctl list-timers
  • Проверьте синтаксис OnCalendar
  • Убедитесь, что сервис существует: systemctl status <имя>.service
  • Проверьте Persistent=true для пропущенных заданий
🛟 Универсальный совет: Если сервис не запускается, попробуйте запустить команду вручную от имени пользователя, указанного в unit-файле: sudo -u www-data /path/to/command. Это покажет ошибки, которые systemd не логирует.

✅ Задания

Базовые задания

  • Создай скрипт /usr/local/bin/hello.sh, который пишет дату в /tmp/hello.log каждые 10 секунд
  • Создай unit-файл /etc/systemd/system/hello.service для этого скрипта
  • Активируй сервис: daemon-reload, start, enable
  • Проверь статус: systemctl status hello
  • Посмотри логи: journalctl -u hello -n 20
  • Настрой автоматический рестарт при падении (Restart=always)
  • Создай таймер, который запускает команду каждый час
  • Посмотри все активные таймеры: systemctl list-timers
  • Останови сервис и проверь, что он остановился
  • Замаскируй сервис (mask) и попробуй запустить — должно не получиться

🧩 Задачи со звёздочкой

  • Создайте сервис для Python/Node.js приложения с правильными настройками безопасности (User, WorkingDirectory)
  • Настройте таймер с RandomizedDelaySec и Persistent=true
  • Создайте сервис, который зависит от другого сервиса (After=, Wants=)
  • Найдите все упавшие сервисы: systemctl --failed и попробуйте их исправить
  • Создайте timer, который запускает бэкап базы данных каждый день в 3:00 ночи
  • Настройте сервис с ограничением ресурсов (MemoryLimit=, CPUQuota=)
