🔍 Мышление сисадмина

Траблшутинг — это не магия

Траблшутинг — это методичный перебор гипотез. Каждый симптом ведёт к набору команд, которые сужают круг подозреваемых. Опытный админ отличается от новичка не знаниями команд, а алгоритмом мышления.

  • Системность — не прыгать между командами, а идти по шагам
  • 📊 Наблюдательность — замечать детали в выводе команд
  • 🧠 Логика — от общего к частному, от симптома к причине
  • Документирование — записывать, что делал и что получилось
Аналогия: Траблшутинг — как медицинская диагностика. Пациент жалуется на головную боль (симптом). Врач не сразу назначает таблетки, а задаёт вопросы, делает анализы, исключает варианты. Так же и админ: пользователь говорит "сервер тормозит" — админ собирает метрики, ищет причину, устраняет.

🧭 Методология: 5 шагов диагностики

Универсальный алгоритм

Этот алгоритм работает для 90% проблем. Запомни его — и ты никогда не будешь в тупике.

  1. Воспроизвести проблему — убедиться, что она реальна, а не "мне показалось"
  2. 📋 Собрать симптомы — логи, метрики, вывод команд. Не гадать, а смотреть факты
  3. 💡 Выдвинуть гипотезы — что может быть причиной? Обычно 2-3 варианта
  4. Проверить гипотезы — по одной, от самой вероятной к менее вероятной
  5. ✅ Исправить и документировать — устранить причину, записать решение

Чек-лист "С чего начать, когда всё плохо"

Если ты зашёл на сервер и "всё тормозит" — выполни эти команды по порядку. Они дадут общую картину за 2 минуты:

# 1. Кто нагружает CPU? top -bn1 | head -20 # 2. Сколько памяти свободно? free -h # 3. Хватает ли места на диске? df -h # 4. Хватает ли inodes? df -i # 5. Какие сервисы упали? systemctl --failed # 6. Что в логах ядра? dmesg -T | tail -30 # 7. Есть ли сеть? ping -c 2 8.8.8.8
💡 Совет: Сохрани этот чек-лист как скрипт /usr/local/bin/healthcheck.sh и запускай при первом входе на незнакомый сервер.

📊 Кейс 1: Сервер внезапно начал тормозить

Симптомы

Пользователи жалуются, что всё медленно. Вы заходите по SSH и видите, что команды выполняются с задержкой. top показывает load average 8.5 при 4 ядрах.

Расследование

Шаг 1: Общая картина — top или htop. Смотрим load average и какой процесс жрёт CPU.

Шаг 2: Детали — ps aux --sort=-%cpu | head -5 — топ-5 процессов по CPU.

Шаг 3: Что это за процесс? — ls -l /proc/<PID>/exe — где лежит исполняемый файл.

Шаг 4: Действие — если процесс не системный, можно kill -9 PID (если это не сломает сервис).

top ps aux --sort=-%cpu | head -5 ls -l /proc/1234/exe kill -9 1234
🎯 Вывод: Нагрузка чаще всего создаётся взбесившимся скриптом или майнером. Быстрое решение — найти и убить. Долгосрочное — разобраться, как он попал на сервер.

💾 Кейс 2: На диске закончилось место

Симптомы

Сервисы падают с ошибкой «No space left on device», невозможно создавать файлы. apt install не работает.

Расследование

Шаг 1: Где проблема? — df -h — смотрим заполненность разделов.

Шаг 2: Кто съел место? — du -sh /* 2>/dev/null | sort -rh | head -10 — ищем большие папки в корне.

Шаг 3: Внутри папки-подозреваемой — du -sh /var/* | sort -rh | head — сужаем круг.

Шаг 4: Конкретные файлы — find /var -type f -size +100M — ищем файлы >100 МБ.

Шаг 5: Освобождаем — удаляем логи, кэш, старые ядра (через apt autoremove).

df -h du -sh /var/* | sort -rh | head find /var -type f -size +100M sudo apt autoremove sudo journalctl --vacuum-size=100M
️ Важно: Не удаляй файлы через rm, пока не поймёшь, что это! Особенно опасно удалять файлы из /var/lib/mysql или /etc.
🎯 Вывод: Чаще всего место заканчивается из-за логов, которые забыли ротировать, или из-за кэша пакетного менеджера. Решение — настроить logrotate и регулярно чистить кэш.

Кейс 3: Сервис не запускается

Симптомы

systemctl start nginx молча падает или выдаёт ошибку. Сайт не открывается. systemctl status nginx показывает "failed".

Расследование

Шаг 1: Статус сервиса — systemctl status nginx — читаем последние сообщения.

Шаг 2: Логи сервиса — journalctl -u nginx --since "5 min ago" — смотрим логи за последние 5 минут.

Шаг 3: Проверка конфига — nginx -t (или аналог для другого сервиса) — валидация конфигурации.

Шаг 4: Порты — ss -tulpn | grep :80 — не занят ли порт другим процессом?

Шаг 5: Исправляем — правим конфиг через nano, перезапускаем.

systemctl status nginx journalctl -u nginx --since "5 min ago" nginx -t ss -tulpn | grep :80 sudo nano /etc/nginx/nginx.conf sudo systemctl restart nginx
🎯 Вывод: 90% проблем — опечатка в конфиге или занятый порт. Логи всегда указывают на причину. Никогда не гадай — читай journalctl.

🌐 Кейс 4: Нет сети / DNS не работает

Симптомы

Сервер не отвечает на пинг, или пинг работает, но curl ya.ru выдаёт "Could not resolve host". Сайт недоступен из интернета.

Расследование

Шаг 1: Есть ли интернет? — ping -c 4 8.8.8.8 (пинг по IP, без DNS).

Шаг 2: Работает ли DNS? — ping -c 4 ya.ru или dig google.com +short.

Шаг 3: Какой DNS настроен? — cat /etc/resolv.conf.

Шаг 4: Какой IP у сервера? — ip a и ip r (маршруты).

Шаг 5: Не блокирует ли фаервол? — sudo ufw status или sudo iptables -L.

ping -c 4 8.8.8.8 ping -c 4 ya.ru cat /etc/resolv.conf ip a ip r sudo ufw status
⚠️ Частая ошибка: Если ping 8.8.8.8 работает, а ping ya.ru нет — проблема в DNS, не в сети. Проверь /etc/resolv.conf и попробуй dig @8.8.8.8 ya.ru.
Вывод: Сетевые проблемы делятся на три категории: нет маршрута (ip r), нет DNS (resolv.conf), блокирует фаервол (ufw/iptables). Определи категорию — и решение найдётся быстро.

dmesg и /proc: глубокая диагностика

dmesg — сообщения ядра

Ядро Linux пишет важные события в кольцевой буфер. Там видны ошибки оборудования, OOM killer, проблемы с дисками.

dmesg -T | tail -50 # последние 50 записей с читаемыми датами dmesg -T | grep -i error # только ошибки dmesg -T | grep -i "oom" # OOM killer (убийца при нехватке памяти) dmesg -T | grep -i "ext4" # ошибки файловой системы journalctl -k # альтернатива через journalctl
⚠️ OOM Killer: Если в dmesg есть Out of memory: Killed process — системе не хватает RAM, и ядро убивает процессы. Нужно добавить RAM или оптимизировать приложения.

/proc — окно в ядро

/proc — виртуальная файловая система. Каждый файл показывает состояние системы в реальном времени.

cat /proc/cpuinfo # информация о CPU (модель, ядра) cat /proc/meminfo # детальная информация о памяти cat /proc/loadavg # load average (как uptime, но только числа) cat /proc/uptime # сколько секунд система работает cat /proc/net/dev # сетевые интерфейсы и трафик cat /proc/PID/status # статус конкретного процесса ls -l /proc/PID/fd # открытые файловые дескрипторы процесса
💡 Зачем? Если процесс ведёт себя странно — /proc/PID/fd покажет, какие файлы он держит открытыми. Если процесс съел всю память — /proc/PID/status покажет сколько.

🎯 Реальные сценарии

Сценарий 1: "Утром зашёл на сервер — всё висит"

# 1. Быстрый healthcheck top -bn1 | head -20 free -h df -h systemctl --failed # 2. Нашли: nginx упал, load average 12 # 3. Смотрим почему nginx упал journalctl -u nginx --since "1 hour ago" # 4. Нашли: "bind() to 0.0.0.0:80 failed (98: Address already in use)" # 5. Кто занял порт 80? ss -tulpn | grep :80 # 6. Нашли: apache2 слушает порт 80 # 7. Решение: остановить apache2, запустить nginx sudo systemctl stop apache2 sudo systemctl start nginx

Сценарий 2: "Сайт медленно грузится"

# 1. Проверяем сеть ping -c 4 ya.ru curl -o /dev/null -s -w "%{time_total}\n" https://ya.ru # 2. Проверяем нагрузку top -bn1 | head -20 iostat -x 1 5 # 3. Проверяем диски df -h df -i # 4. Проверяем MySQL (если есть) mysqladmin -u root -p status # 5. Смотрим slow query log sudo tail -50 /var/log/mysql/mysql-slow.log

🔧 Troubleshooting: Частые проблемы

❓ "Команды выполняются медленно"
  • Проверьте load average: uptime
  • Проверьте I/O: iostat -x 1 — если %util = 100%, диск перегружен
  • Проверьте память: free -h — если swap используется, не хватает RAM
  • Найдите виновника: ps aux --sort=-%cpu | head
❓ "No space left on device", но df показывает место
  • Проверьте inodes: df -i
  • Если inodes = 100% — слишком много мелких файлов
  • Найдите: find / -xdev -printf '%h\n' | sort | uniq -c | sort -rn | head
  • Удалите ненужные файлы (сессии, временные файлы, кэш)
❓ "Сервис падает сразу после запуска"
  • Смотрите логи: journalctl -u <сервис> -n 50
  • Проверьте конфиг: nginx -t или аналог
  • Проверьте права на файлы: ls -l /path/to/files
  • Проверьте, не занят ли порт: ss -tulpn | grep :PORT
  • Попробуйте запустить вручную: sudo -u www-data /usr/bin/nginx
❓ "Нет сети после настройки"
  • Проверьте интерфейс: ip a — есть ли IP?
  • Проверьте маршруты: ip r — есть ли default?
  • Проверьте DNS: cat /etc/resolv.conf
  • Проверьте фаервол: sudo ufw status
  • Попробуйте пинг по IP: ping 8.8.8.8
❓ "Логи не пишутся"
  • Проверьте место на диске: df -h
  • Проверьте inodes: df -i
  • Проверьте права на файл лога: ls -l /var/log/...
  • Проверьте, запущен ли сервис логирования: systemctl status rsyslog
🛟 Универсальный совет: Если не знаешь, с чего начать — выполни чек-лист из раздела "Методология". Он покрывает 90% случаев.

✅ Задания

Базовые задания

  • Запусти stress -c 1 (если установлен) и найди его через top, потом убей через kill
  • Создай большой файл dd if=/dev/zero of=bigfile bs=1M count=500 в /tmp, найди его через du и удали
  • Останови ssh, измени порт в конфиге на занятый (например, 80, если nginx слушает), попробуй стартовать и проанализируй ошибку
  • Выполни чек-лист "С чего начать" на своём сервере и запиши результаты
  • Найди все файлы больше 100МБ в /var: find /var -type f -size +100M
  • Посмотри сообщения ядра за последний час: dmesg -T | tail -30
  • Проверь inodes: df -i — сколько свободно?
  • Найди процесс, который держит открытым файл /var/log/syslog: lsof /var/log/syslog

🧩 Задачи со звёздочкой

  • Создайте скрипт /usr/local/bin/healthcheck.sh, который выполняет чек-лист из раздела "Методология" и выводит результаты в удобном виде
  • Найдите процесс, который использует больше всего открытых файлов: проверьте /proc/PID/fd для топ-5 процессов по памяти
  • Сымитируйте нехватку памяти: запустите stress --vm 2 --vm-bytes 512M и смотрите, как OOM killer убивает процессы через dmesg -T
