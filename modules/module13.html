🚀 Контейнеризация

Что такое Docker?

Docker — платформа для упаковки, доставки и запуска приложений в изолированных контейнерах. Контейнер содержит всё необходимое: код, библиотеки, зависимости — и работает одинаково где угодно.

  • 📦 Изоляция — каждое приложение в своём контейнере, не мешает другим
  • ⚡ Скорость — контейнеры запускаются за секунды, не минуты
  • 🔄 Повторяемость — "работает на моей машине" больше не проблема
  • Масштабирование — легко запустить 10 копий приложения

️ Контейнер vs Виртуальная машина

В чём разница?

Виртуальная машина (VM)          Контейнер (Docker)
┌──────────────────────┐         ┌──────────────────────┐
│      Приложение       │         │      Приложение       │
──────────────────────┤         ├──────────────────────┤
│   Библиотеки/Зависимости│       │   Библиотеки/Зависимости│
├──────────────────────┤         ├──────────────────────┤
│    Гостевая ОС         │         │                      │
├──────────────────────┤         │                      │
│    Гипервизор          │         ├──────────────────────┤
├──────────────────────┤         │   Docker Engine       │
│      Хостовая ОС       │         ├──────────────────────┤
──────────────────────┤         │      Хостовая ОС       │
└──────────────────────┘         └──────────────────────┘
        Тяжёлая (ГБ)                     Лёгкий (МБ)
        Запуск: минуты                   Запуск: секунды
        Полная изоляция                  Общее ядро ОС
Когда что использовать:
  • Контейнеры — микросервисы, веб-приложения, CI/CD, быстрое развёртывание
  • ВМ — разные ОС (Windows + Linux), полная изоляция, legacy-приложения

📦 Установка Docker

Linux (Ubuntu/Debian) или WSL (Windows)

Рекомендуется установить Docker Engine внутри вашего Linux-окружения (включая WSL). Это даст полный контроль и не требует Docker Desktop.

# Удалите старые версии (если есть) sudo apt remove docker docker-engine docker.io containerd runc # Установите необходимые пакеты sudo apt update sudo apt install -y ca-certificates curl gnupg lsb-release # Добавьте официальный GPG-ключ Docker sudo mkdir -p /etc/apt/keyrings curl -fsSL https://download.docker.com/linux/ubuntu/gpg | sudo gpg --dearmor -o /etc/apt/keyrings/docker.gpg # Добавьте репозиторий echo "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.gpg] https://download.docker.com/linux/ubuntu $(lsb_release -cs) stable" | sudo tee /etc/apt/sources.list.d/docker.list > /dev/null # Установите Docker Engine sudo apt update sudo apt install -y docker-ce docker-ce-cli containerd.io docker-compose-plugin # Добавьте своего пользователя в группу docker sudo usermod -aG docker $USER
⚠️ Важно: Выйдите из системы и зайдите снова (или перезапустите WSL), чтобы изменения в группе вступили в силу. Иначе придётся использовать sudo docker.
# Проверьте установку docker run hello-world

️ Работа с образами

Что такое образ?

Образ (image) — это шаблон контейнера. Как класс в ООП: из одного образа можно создать много контейнеров. Образы хранятся в реестрах (Docker Hub, GitHub Container Registry).

# Скачать образ из Docker Hub docker pull nginx:latest docker pull nginx:1.21 # конкретная версия (рекомендуется!) docker pull postgres:15-alpine # лёгкая версия на Alpine Linux # Посмотреть локальные образы docker images docker image ls # Удалить образ docker rmi nginx:latest # Найти образ на Docker Hub docker search nginx # Очистка неиспользуемых образов docker image prune # удалить "висячие" образы docker system prune -a # удалить всё неиспользуемое (осторожно!)
💡 Тегирование: Тег — это версия образа. nginx:latest — последняя, nginx:1.21 — конкретная. Всегда используйте конкретные версии в production! latest может сломать всё после обновления.

Свои теги и публикация

# Добавить тег к образу docker tag myapp:1.0 myrepo/myapp:1.0 docker tag myapp:1.0 myrepo/myapp:latest # Войти в реестр docker login # Отправить образ docker push myrepo/myapp:1.0 docker push myrepo/myapp:latest

📦 Контейнеры

Запуск и управление

# Запустить контейнер docker run hello-world docker run -d --name web -p 8080:80 nginx # -d = в фоне (detached) # --name = имя контейнера # -p 8080:80 = проброс порта (хост:контейнер) # Посмотреть запущенные контейнеры docker ps docker ps -a # все, включая остановленные # Остановить/запустить/перезапустить docker stop web docker start web docker restart web # Удалить контейнер docker rm web docker rm -f web # принудительно (если запущен) # Посмотреть логи контейнера docker logs web docker logs -f web # следить в реальном времени docker logs --tail 50 web # последние 50 строк

Вход в контейнер и мониторинг

# Войти в запущенный контейнер docker exec -it web bash docker exec -it web sh # если bash нет (Alpine) # Посмотреть ресурсы docker stats # мониторинг в реальном времени docker stats web # только один контейнер # Детальная информация docker inspect web # JSON со всей информацией docker inspect --format='{{.State.Status}}' web # только статус
Флаги exec: -i = интерактивный, -t = псевдо-TTY. Вместе -it дают полноценный терминал внутри контейнера.

🛠️ Dockerfile

Создание своего образа

Dockerfile — инструкция по сборке образа. Каждая команда создаёт слой (layer).

# Пример: Dockerfile для Node.js приложения FROM node:18-alpine # базовый образ (конкретная версия!) WORKDIR /app # рабочая директория внутри контейнера COPY package*.json ./ # копируем файлы зависимостей RUN npm install # устанавливаем зависимости COPY . . # копируем весь код EXPOSE 3000 # какой порт слушает приложение CMD ["node", "app.js"] # команда запуска
🧠 Команды Dockerfile:
  • FROM — базовый образ (всегда первая команда)
  • WORKDIR — рабочая директория
  • COPY — копировать файлы с хоста
  • RUN — выполнить команду при сборке
  • EXPOSE — документация, какой порт открыт
  • CMD — команда запуска контейнера
  • ENV — переменные окружения

Сборка и запуск

# Собрать образ docker build -t myapp:1.0 . # -t = тег (имя:версия) # . = текущая директория (где Dockerfile) # Запустить docker run -d --name myapp -p 3000:3000 myapp:1.0 # Проверить docker ps curl localhost:3000

🎼 Docker Compose

Зачем Compose?

Когда приложение состоит из нескольких контейнеров (веб + БД + кэш), запускать их вручную неудобно. Compose описывает всю инфраструктуру в одном файле.

# docker-compose.yml version: '3.8' services: web: image: nginx:1.21 ports: - "8080:80" volumes: - ./html:/usr/share/nginx/html depends_on: - app app: build: . ports: - "3000:3000" environment: - DB_HOST=db - DB_PASSWORD=secret depends_on: - db db: image: postgres:15-alpine environment: - POSTGRES_PASSWORD=secret volumes: - pgdata:/var/lib/postgresql/data volumes: pgdata:
🧠 Ключевые поля:
  • image — готовый образ
  • build — собрать из Dockerfile
  • ports — проброс портов
  • volumes — постоянные данные
  • environment — переменные окружения
  • depends_on — порядок запуска

Управление Compose

# Запустить всё docker compose up docker compose up -d # в фоне # Посмотреть статус docker compose ps # Посмотреть логи docker compose logs docker compose logs -f web # только web, в реальном времени # Остановить всё docker compose stop docker compose down # остановить и удалить контейнеры docker compose down -v # + удалить volumes (осторожно!)

💾 Volumes и сети

Хранение данных: volumes

Контейнеры эфемерны — данные внутри теряются при удалении. Volumes решают эту проблему.

# Создать volume docker volume create pgdata # Использовать в контейнере docker run -d --name db -v pgdata:/var/lib/postgresql/data postgres # Посмотреть volumes docker volume ls docker volume inspect pgdata # Удалить docker volume rm pgdata docker volume prune # удалить неиспользуемые
💡 Bind mount vs Volume:
  • -v pgdata:/path — volume (управляется Docker, надёжнее)
  • -v ./local:/path — bind mount (прямая папка хоста, удобно для разработки)

Связка контейнеров: network

# Создать сеть docker network create mynet # Запустить контейнеры в сети docker run -d --name app --net mynet myapp docker run -d --name redis --net mynet redis # Контейнеры видят друг друга по именам! # Из app можно обратиться к redis:6379 # Посмотреть сети docker network ls docker network inspect mynet
💡 DNS в Docker: Контейнеры в одной сети резолвят друг друга по имени. Не нужны IP-адреса!

Отладка контейнеров

Типичный процесс отладки

# 1. Контейнер не запускается — смотрим логи docker logs myapp docker logs --tail 100 myapp # 2. Контейнер запущен, но не работает — входим внутрь docker exec -it myapp bash # Внутри контейнера: ps aux netstat -tulpn cat /app/logs/app.log # 3. Смотрим ресурсы docker stats myapp # 4. Детальная информация docker inspect myapp | jq '.[0].State' docker inspect myapp | jq '.[0].Config.Env'

⭐ Best practices

1. Multi-stage builds (уменьшение размера)

Собираем в одном образе, запускаем в другом. Итоговый образ в 10 раз меньше!

# Stage 1: сборка FROM node:18 AS builder WORKDIR /app COPY . . RUN npm install && npm run build # Stage 2: запуск (только результат!) FROM node:18-alpine WORKDIR /app COPY --from=builder /app/dist ./dist COPY --from=builder /app/node_modules ./node_modules CMD ["node", "dist/main.js"]

2. .dockerignore

Исключайте ненужные файлы из сборки (как .gitignore):

# .dockerignore node_modules .git .env *.log Dockerfile docker-compose.yml

3. Не запускать от root

FROM node:18-alpine RUN adduser -D appuser USER appuser WORKDIR /home/appuser/app # ... остальной Dockerfile
⚠️ Безопасность: Если контейнер взломают, злоумышленник получит root на хосте (если не использовать USER).

4. Конкретные версии образов

# ❌ Плохо: FROM node:latest FROM nginx # ✅ Хорошо: FROM node:18.17-alpine FROM nginx:1.25.2

5. Один процесс на контейнер

Не пытайтесь запустить nginx + php-fpm + cron в одном контейнере. Разделите на отдельные контейнеры и свяжите через Compose.

Реальные сценарии

Сценарий 1: Деплой Node.js + PostgreSQL

# 1. Создаём docker-compose.yml (см. раздел Compose) # 2. Запускаем docker compose up -d # 3. Проверяем docker compose ps docker compose logs -f app # 4. Обновление приложения docker compose build app docker compose up -d app # 5. Остановка docker compose down

Сценарий 2: Быстрый запуск инструментов

# Nginx за 10 секунд docker run -d -p 80:80 nginx # Redis для кэша docker run -d -p 6379:6379 redis:alpine # PostgreSQL для разработки docker run -d -p 5432:5432 -e POSTGRES_PASSWORD=dev postgres:15-alpine # MongoDB docker run -d -p 27017:27017 mongo:7 # Всё без установки на хост!

🔧 Troubleshooting: Если не получилось

❓ "Container exited immediately" (контейнер падает сразу)
  • Смотрите логи: docker logs myapp
  • Частые причины: ошибка в приложении, нет переменных окружения, неправильный CMD
  • Запустите интерактивно для отладки: docker run -it myapp sh
❓ "Port already in use" (порт занят)
  • Проверьте, кто занял порт: ss -tulpn | grep :8080
  • Остановите конфликтующий сервис или используйте другой порт: -p 8081:80
  • Или удалите старый контейнер: docker rm -f web
"No space left on device"
  • Docker съел всё место образами и контейнерами
  • Очистка: docker system prune -a (удалит всё неиспользуемое)
  • Или мягче: docker image prune (только висячие образы)
  • Проверьте размер: docker system df
❓ "Permission denied" внутри контейнера
  • Контейнер запущен от root, а файлы принадлежат другому пользователю
  • Добавьте в Dockerfile: USER appuser
  • Или запустите с флагом: docker run --user $(id -u):$(id -g) ...
❓ "Контейнер не видит файлы" (volumes)
  • Проверьте пути: docker inspect myapp | jq '.[0].Mounts'
  • Bind mount: убедитесь, что папка на хосте существует
  • Volume: данные сохраняются между перезапусками контейнера
  • Права: docker exec myapp ls -la /path
❓ "Cannot connect to Docker daemon"
  • Docker не запущен: sudo systemctl start docker
  • Пользователь не в группе docker: sudo usermod -aG docker $USER и перелогиниться
  • Или используйте sudo docker временно
🛟 Универсальный совет: Если что-то не работает — docker logs в 90% случаев показывает причину. Если контейнер упал — docker logs myapp. Если не запускается — docker run -it myapp sh и смотрите внутри.

✅ Задания

Базовые задания

  • Установи Docker согласно инструкции для своей ОС
  • Запусти контейнер hello-world и убедись, что всё работает
  • Запусти nginx с пробросом порта 8080: docker run -d -p 8080:80 nginx, проверь в браузере
  • Посмотри все запущенные контейнеры: docker ps
  • Посмотри логи nginx: docker logs <container_id>
  • Запусти PostgreSQL с volume, удали контейнер, запусти заново — проверь, что данные сохранились
  • Создай пользовательскую сеть и запусти в ней два контейнера, проверь связь между ними
  • Напиши простой Dockerfile для Python-приложения (hello world), собери и запусти

Задачи со звёздочкой

  • Создайте multi-stage Dockerfile для Node.js приложения: сборка в одном образе, запуск в другом (alpine)
  • Напишите docker-compose.yml с тремя сервисами: nginx (reverse proxy), Node.js app, PostgreSQL. Настройте сети и volumes
  • Настройте .dockerignore для проекта, соберите образ до и после — сравните размер через docker images
  • Создайте Dockerfile, который не запускается от root (добавьте USER), проверьте через docker exec -it myapp whoami
