🗄️ Данные — это всё

Зачем админу управление хранилищем?

Сервер без данных бесполезен. Администратор должен уметь добавлять диски, расширять разделы без простоя, следить за здоровьем железа и восстанавливать данные после сбоев.

  • Управление дисками — разметка, форматирование, монтирование
  • 🧩 LVM — гибкое управление томами (расширение на лету)
  • 🏥 Здоровье дисков — предсказание отказов через SMART
  • 🛡️ Надёжность — правильное монтирование, inodes, fstab
💡 Аналогия: Жёсткий диск — это большой пустой склад. Разметка (partitioning) — это деление склада на комнаты. Файловая система (formatting) — это стеллажи и учётная книга. Монтирование — это открытие дверей в эти комнаты.

📁 Файловые системы

Какую ФС выбрать?

Файловая система определяет, как данные хранятся на диске. В Linux есть несколько стандартов:

┌──────────┬──────────────────────────────────────────────┐
│ ext4     │ Стандарт для Linux. Надёжная, быстрая,       │
│          │ журналируемая. Идеальна для системных дисков.│
├────────────────────────────────────────────────────────┤
│ xfs      │ Отлична для больших файлов и высокой         │
│          │ параллельной нагрузки. Стандарт для RHEL.    │
├──────────┼──────────────────────────────────────────────┤
│ btrfs    │ Современная: снапшоты, сжатие, рейды.        │
│          │ Сложнее, но мощнее.                           │
└──────────┴──────────────────────────────────────────────┘
🧠 Правило: Если не знаешь, что выбрать — бери ext4. Для баз данных с огромными файлами — xfs. Для домашних серверов с бэкапами — btrfs.

✂️ Разметка и форматирование

Просмотр дисков

lsblk # дерево дисков и разделов lsblk -f # с информацией о файловых системах sudo fdisk -l # детальная информация о разделах sudo blkid # UUID и типы всех разделов

Создание раздела и форматирование

# 1. Разметка диска (интерактивно) sudo fdisk /dev/sdb # Внутри fdisk: n (новый раздел), p (primary), Enter (по умолчанию), w (записать) # 2. Форматирование в ext4 sudo mkfs.ext4 /dev/sdb1 # 3. Форматирование в xfs sudo mkfs.xfs /dev/sdb1
⚠️ Опасно: Команда mkfs уничтожает все данные на разделе. Всегда проверяй, тот ли диск ты выбрал (lsblk).

🔗 Монтирование и fstab

Ручное монтирование

Чтобы использовать раздел, его нужно «примонтировать» к дереву каталогов.

# Создать точку монтирования sudo mkdir -p /mnt/data # Примонтировать раздел sudo mount /dev/sdb1 /mnt/data # Проверить, что смонтировано df -h | grep sdb1 # Размонтировать (когда никто не использует файлы) sudo umount /mnt/data

Автоматическое монтирование: /etc/fstab

Чтобы диск монтировался при загрузке, добавь запись в /etc/fstab. Используй UUID, а не /dev/sdb1 (имя диска может измениться после перезагрузки)!

# 1. Узнать UUID раздела sudo blkid /dev/sdb1 # Вывод: UUID="12345678-1234-1234-1234-123456789abc" # 2. Открыть fstab sudo nano /etc/fstab # 3. Добавить строку: UUID=12345678-1234-1234-1234-123456789abc /mnt/data ext4 defaults 0 2
Формат fstab: UUID точка_монтирования тип_ФС опции dump pass
  • defaults — стандартные опции (rw, suid, dev, exec, auto, nouser, async)
  • 0 — не делать бэкап через dump
  • 2 — проверять ФС при загрузке (0 для swap, 1 для root)
🚨 КРИТИЧНО: Ошибка в /etc/fstab может привести к тому, что система не загрузится (попадёт в emergency mode). После редактирования всегда проверяй: sudo mount -a. Если ошибок нет — всё хорошо.

🧩 LVM: Логический менеджер томов

Зачем нужен LVM?

Обычные разделы фиксированы. LVM позволяет создавать «резиновые» тома, которые можно расширять на лету, добавлять новые диски в пул и делать снапшоты.

Физические диски (PV)      Группа томов (VG)      Логические тома (LV)
┌──────────┐               ┌──────────────────┐   ┌──────────────┐
│ /dev/sda │──┐            │                  │──▶│ /dev/vg0/lv1 │ (root)
──────────┘  │            │     vg0          │   └──────────────┘
┌──────────┐  │            │  (20 GB pool)    │   ┌──────────────┐
│ /dev/sdb │──            │                  │──▶│ /dev/vg0/lv2 │ (data)
──────────┘               └──────────────────   └──────────────┘

Создание LVM с нуля

# 1. Создать физический том (Physical Volume) sudo pvcreate /dev/sdb # 2. Создать группу томов (Volume Group) sudo vgcreate vg_data /dev/sdb # 3. Создать логический том (Logical Volume) размером 10G sudo lvcreate -L 10G -n lv_data vg_data # 4. Отформатировать и смонтировать sudo mkfs.ext4 /dev/vg_data/lv_data sudo mkdir -p /mnt/data sudo mount /dev/vg_data/lv_data /mnt/data

Расширение тома на лету (без простоя!)

# 1. Добавить новый диск в группу томов sudo pvcreate /dev/sdc sudo vgextend vg_data /dev/sdc # 2. Расширить логический том на все свободное место sudo lvextend -l +100%FREE /dev/vg_data/lv_data # 3. Расширить файловую систему (онлайн!) # Для ext4: sudo resize2fs /dev/vg_data/lv_data # Для xfs: sudo xfs_growfs /mnt/data
💡 Магия LVM: Пользователи и сервисы даже не заметят, что диск расширился. Никаких перезагрузок!

🏥 Мониторинг здоровья дисков (SMART)

S.M.A.R.T. — система самодиагностики

Современные диски имеют встроенную систему мониторинга. Она может предупредить об отказе до того, как диск умрёт.

# Установить утилиты sudo apt install -y smartmontools # Посмотреть полное состояние диска sudo smartctl -a /dev/sda # Краткий тест здоровья sudo smartctl -H /dev/sda # Запустить короткий тест (не нагружает диск) sudo smartctl -t short /dev/sda # Посмотреть результаты тестов sudo smartctl -l selftest /dev/sda
⚠️ Критичные атрибуты: В выводе smartctl -a ищи строки Reallocated_Sector_Ct, Current_Pending_Sector, UDMA_CRC_Error_Count. Если их значение (RAW_VALUE) > 0 — диск начинает сыпаться, срочно меняй!

Реальные сценарии

Сценарий 1: Добавление нового диска на production-сервер

# 1. Находим новый диск lsblk # Видим sdb без разделов # 2. Создаём раздел на весь диск sudo fdisk /dev/sdb # n -> p -> Enter -> Enter -> w # 3. Форматируем в ext4 sudo mkfs.ext4 /dev/sdb1 # 4. Узнаём UUID sudo blkid /dev/sdb1 # 5. Создаём папку и монтируем sudo mkdir -p /mnt/backups sudo mount /dev/sdb1 /mnt/backups # 6. Добавляем в fstab (ОБЯЗАТЕЛЬНО через UUID!) echo "UUID=ваш-uuid /mnt/backups ext4 defaults 0 2" | sudo tee -a /etc/fstab # 7. Проверяем fstab sudo mount -a df -h | grep backups

Сценарий 2: Расширение LVM тома, когда кончилось место

# 1. Проблема: df -h показывает 100% на /dev/mapper/vg0-lv_data # 2. Добавили новый диск (sdc) в сервер (горячая замена или через панель хостинга) # 3. Расширяем LVM sudo pvcreate /dev/sdc sudo vgextend vg0 /dev/sdc sudo lvextend -l +100%FREE /dev/mapper/vg0-lv_data sudo resize2fs /dev/mapper/vg0-lv_data # 4. Проверяем df -h | grep lv_data # Готово! Место добавлено без перезагрузки.

🔧 Troubleshooting: Если не получилось

Система не грузится, попадает в "emergency mode"
  • Причина: ошибка в /etc/fstab (неправильный UUID, нет диска)
  • Введи пароль root
  • Перемонтируй root в запись: mount -o remount,rw /
  • Исправь fstab: nano /etc/fstab (закомментируй проблемную строку через #)
  • Перезагрузись: reboot
❓ "Read-only filesystem" — нельзя создать файл
  • ФС перемонтировалась в read-only из-за ошибки диска
  • Проверь ошибки: dmesg -T | grep -i ext4 или smartctl -a /dev/sda
  • Временное решение: sudo mount -o remount,rw /mnt/data
  • Если диск сыпется — срочно копируй данные и меняй диск!
"No space left on device", но df -h показывает место
  • Закончились inodes (слишком много мелких файлов)
  • Проверь: df -i
  • Найди папку с кучей файлов: find / -xdev -printf '%h\n' | sort | uniq -c | sort -rn | head
  • Часто виноваты: сессии PHP, кэш, очереди почты
❓ "Device is busy" при umount
  • Какой-то процесс использует файлы на диске
  • Найди кто: lsof +D /mnt/data или fuser -m /mnt/data
  • Останови процесс или смени директорию (cd ~) и повтори umount
🛟 Универсальный совет: Перед любой работой с дисками (особенно fdisk, mkfs) трижды проверь имя диска через lsblk. Ошибка здесь стоит потерянных данных.

✅ Задания

Базовые задания

  • Просмотри все диски и разделы: lsblk и lsblk -f
  • Проверь использование inodes на всех разделах: df -i
  • Узнай UUID всех разделов: sudo blkid
  • Создай файл размером 100МБ: dd if=/dev/zero of=/tmp/testfile bs=1M count=100 и проверь его размер через ls -lh
  • Посмотри SMART-статус основного диска: sudo smartctl -H /dev/sda (если smartctl установлен)
  • Найди, какие процессы держат открытыми файлы в /var/log: sudo lsof +D /var/log
  • Проверь fstab на ошибки (сухой прогон): sudo mount -a
  • Посмотри историю монтирования: cat /etc/mtab или findmnt

🧩 Задачи со звёздочкой

  • Создайте LVM том из двух виртуальных дисков (если есть возможность добавить диски в ВМ), расширьте его на лету и проверьте через df -h
  • Настройте автоматический SMART-тест через cron: короткий тест каждый день в 2:00, длинный — каждое воскресенье в 3:00
  • Сымитируйте ошибку fstab (добавьте неверный UUID), перезагрузите ВМ, войдите в emergency mode и исправьте ошибку через mount -o remount,rw /
  • Напишите скрипт, который проверяет заполненность всех разделов и отправляет алерт в Telegram, если какой-то раздел заполнен > 85%
