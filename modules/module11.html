👀 Держи руку на пульсе

Зачем мониторинг?

Мониторинг — это система раннего предупреждения. Без неё вы узнаёте о проблеме, когда клиенты уже жалуются. С мониторингом — вы узнаёте до того, как проблема станет критичной.

  • 📊 Метрики — числа, описывающие состояние системы (CPU, RAM, диск)
  • 📈 Визуализация — графики, чтобы видеть тренды
  • 🔔 Алерты — уведомления, когда метрики выходят за пределы
💡 Три уровня мониторинга:
  1. Ручной — вы заходите и смотрите top, df, free
  2. Скрипты + cron — автоматическая проверка каждые N минут + алерт в Telegram
  3. Система мониторинга — Prometheus + Grafana, графики, история, алерты

📊 Системные метрики

uptime и load average

uptime # 14:23:15 up 45 days, 3:12, 2 users, load average: 0.15, 0.22, 0.18
🧠 Что значат 3 числа load average?
Средняя загрузка за 1 минуту, 5 минут, 15 минут.

Правило: Сравнивайте с количеством ядер CPU. Если у вас 4 ядра:
  • Load < 4 — всё хорошо
  • Load = 4 — полная загрузка
  • Load > 4 — процессы ждут в очереди (система тормозит)
Узнать количество ядер: nproc

free: оперативная память

free -h # total used free shared buff/cache available # Mem: 7.8G 2.1G 1.2G 156M 4.5G 5.3G # Swap: 2.0G 0B 2.0G
🧠 Важная колонка — available, а не free!
Linux активно использует свободную память под кэш (buff/cache), но может освободить её мгновенно. Поэтому available — реальное количество доступной памяти.

df и du: диск

df -h # использование дисков df -i # использование inodes (не менее важно!) du -sh /var/* | sort -rh | head -10 # что занимает больше всего в /var
⚠️ Inodes: Даже если df -h показывает свободное место, df -i может показать 100% inodes. Это значит, что закончились «слоты» для файлов (много мелких файлов). Система не сможет создавать новые файлы!

vmstat: виртуальная память и CPU

vmstat 1 # обновление каждую секунду # procs -----------memory---------- ---swap-- -----io---- -system-- ------cpu----- # r b swpd free buff cache si so bi bo in cs us sy id wa st
🧠 Ключевые колонки:
  • r — процессы, ожидающие CPU (если больше ядер — перегрузка)
  • b — процессы в uninterruptible sleep (ждут I/O)
  • wa — процент CPU, потраченный на ожидание I/O (если > 20% — проблема с диском)
  • si/so — swap in/out (если не 0 — не хватает RAM)
  • id — процент простоя CPU (idle)

iostat: детальный I/O диска

sudo apt install sysstat # если не установлен iostat -x 1 # расширенный вывод каждую секунду
🧠 Ключевые метрики:
  • %util — загрузка диска (100% = диск перегружен)
  • await — среднее время ответа (мс). Если > 50мс для SSD — проблема
  • r/s, w/s — операций чтения/записи в секунду

dmesg: сообщения ядра

dmesg -T # сообщения ядра с читаемыми timestamps dmesg -T | tail -50 # последние 50 записей dmesg -T | grep -i error # только ошибки dmesg -T | grep -i "oom" # OOM killer (убийца при нехватке памяти) journalctl -k # альтернатива через journalctl
⚠️ OOM Killer: Если в dmesg есть Out of memory: Killed process — системе не хватает RAM, и ядро убивает процессы. Нужно добавить RAM или оптимизировать приложения.

📂 /proc: информация о системе

Виртуальная файловая система

/proc — это не реальные файлы на диске. Это окно в ядро Linux. Каждый файл показывает состояние системы в реальном времени.

cat /proc/cpuinfo # информация о CPU (модель, ядра, частота) cat /proc/meminfo # детальная информация о памяти cat /proc/loadavg # load average (как uptime, но только числа) cat /proc/uptime # сколько секунд система работает cat /proc/net/dev # сетевые интерфейсы и трафик cat /proc/diskstats # статистика дисков

/proc/PID — информация о процессе

ls /proc/1234/ # что «видит» процесс cat /proc/1234/status # статус процесса (память, потоки) cat /proc/1234/cmdline # командная строка запуска cat /proc/1234/environ # переменные окружения ls -l /proc/1234/exe # путь к исполняемому файлу ls -l /proc/1234/fd # открытые файловые дескрипторы
💡 Зачем? Если процесс ведёт себя странно — /proc/PID/fd покажет, какие файлы он держит открытыми. Если процесс съел всю память — /proc/PID/status покажет сколько.

🔬 Практика: системные утилиты

Пошаговая практика

uptime # load average nproc # количество ядер free -h # память df -h # диски df -i # inodes vmstat 1 # виртуальная память (Ctrl+C для остановки) iostat -x 1 # I/O дисков dmesg -T | tail -20 # последние сообщения ядра cat /proc/cpuinfo | grep "model name" | head -1 # модель CPU cat /proc/loadavg # текущий load

📝 Скрипты мониторинга

Скрипт проверки места на диске

#!/bin/bash THRESHOLD=90 USAGE=$(df / | awk 'NR==2 {print $5}' | sed 's/%//') if [ "$USAGE" -gt "$THRESHOLD" ]; then echo "⚠️ Диск заполнен на $USAGE%!" | wall echo "$(date): Disk alert: $USAGE%" >> /var/log/disk-alert.log fi

Скрипт проверки сервисов

#!/bin/bash SERVICES="nginx ssh mysql" for SERVICE in $SERVICES; do if systemctl is-active --quiet "$SERVICE"; then echo "✅ $SERVICE — работает" else echo "❌ $SERVICE — УПАЛ!" | wall echo "$(date): $SERVICE is down!" >> /var/log/service-alert.log # Попытка перезапуска: sudo systemctl restart "$SERVICE" fi done

Скрипт проверки HTTP

#!/bin/bash URLS="https://google.com https://github.com http://localhost" for URL in $URLS; do STATUS=$(curl -o /dev/null -s -w "%{http_code}" --max-time 5 "$URL") if [ "$STATUS" -eq 200 ]; then echo "✅ $URL — $STATUS OK" else echo "❌ $URL — $STATUS FAIL" fi done

🔔 Алерты в Telegram

Создание бота

  1. Напишите @BotFather в Telegram → /newbot → получите токен
  2. Напишите любое сообщение боту
  3. Получите chat_id: curl -s https://api.telegram.org/bot<TOKEN>/getUpdates | grep chat_id

Отправка уведомления

#!/bin/bash TOKEN="123456:ABC-DEF" CHAT_ID="987654321" send_alert() { local MESSAGE="$1" curl -s -X POST "https://api.telegram.org/bot$TOKEN/sendMessage" \ -d chat_id="$CHAT_ID" \ -d text="$MESSAGE" \ -d parse_mode="Markdown" > /dev/null } # Использование: DISK_USAGE=$(df / | awk 'NR==2 {print $5}' | sed 's/%//') if [ "$DISK_USAGE" -gt 90 ]; then send_alert "⚠️ *Диск заполнен на ${DISK_USAGE}%!* Сервер: $(hostname) Время: $(date)" fi
💡 Совет: Создайте функцию send_alert() один раз и используйте её во всех скриптах. Подключите через source /usr/local/bin/alert.sh.

📈 Prometheus + Grafana: обзор

Что это?

Это стандарт индустрии для мониторинга серверов. Два инструмента работают вместе:

  • Prometheus — собирает и хранит метрики (числа). Опрашивает серверы каждые N секунд.
  • Grafana — рисует красивые графики из данных Prometheus. Создаёт алерты.
  • Node Exporter — агент, который отдаёт метрики сервера (CPU, RAM, диск, сеть) в Prometheus.
┌──────────────┐    pull     ┌───────────────┐
│  Prometheus   │◄────────────│ Node Exporter │ (на каждом сервере)
│  (хранение)   │             │ :9100         │
└──────┬───────┘             └───────────────┘
       │
       ▼
┌──────────────┐
│   Grafana     │  ← Графики, дашборды, алерты
│  (визуализация)│
└──────────────┘

Быстрая установка через Docker

# 1. Node Exporter (собирает метрики): docker run -d --name=node_exporter \ --net="host" \ --pid="host" \ -v "/:/host:ro,rslave" \ prom/node-exporter:latest \ --path.rootfs=/host # 2. Prometheus (хранит): # Создайте prometheus.yml: # global: # scrape_interval: 15s # scrape_configs: # - job_name: 'node' # static_configs: # - targets: ['localhost:9100'] docker run -d --name=prometheus \ -p 9090:9090 \ -v $(pwd)/prometheus.yml:/etc/prometheus/prometheus.yml \ prom/prometheus # 3. Grafana (визуализирует): docker run -d --name=grafana \ -p 3000:3000 \ grafana/grafana-oss # Откройте http://localhost:3000 (admin/admin) # Добавьте Prometheus как Data Source (http://localhost:9090) # Импортируйте Dashboard ID 1860 (Node Exporter Full)
💡 Готовые дашборды: На сайте grafana.com/grafana/dashboards есть тысячи готовых дашбордов. Dashboard ID 1860 — самый популярный для мониторинга Linux-сервера.

Сценарий 2: Настройка полного мониторинга за 10 минут

  2. Добавьте функцию send_alert() с Telegram-ботом
  3. Добавьте скрипт в cron: */5 * * * * /usr/local/bin/monitor.sh

  5. Импортируйте Dashboard 1860 в Grafana

🔧 Troubleshooting: Если не получилось

❓ Высокий load average, но CPU свободен
  • Процессы ждут I/O (диск)
  • Проверьте: vmstat 1 — колонка wa (I/O wait)
  • Проверьте: iostat -x 1 — колонка %util
  • Решение: более быстрый диск (SSD), оптимизация запросов
❓ Памяти «нет», но free показывает много buff/cache
  • Это нормально! Linux кэширует диски в свободную память
  • Смотрите колонку available, а не free
  • Если available мало — проблема реальная
❓ "No space left on device", но df показывает место
  • Проверьте inodes: df -i
  • Если inodes = 100% — слишком много мелких файлов
  • Найдите: find / -xdev -printf '%h\n' | sort | uniq -c | sort -rn | head
  • Удалите ненужные файлы (сесии, временные файлы, кэш)
❓ OOM Killer убивает процессы
  • Проверьте: dmesg -T | grep -i oom
  • Системе не хватает RAM
  • Найдите, кто ест память: ps aux --sort=-%mem | head
  • Решение: добавить RAM, swap, или оптимизировать приложение
❓ Алерт в Telegram не приходит
  • Проверьте токен и chat_id: curl https://api.telegram.org/botTOKEN/getMe
  • Убедитесь, что написали боту хотя бы одно сообщение
  • Проверьте, что cron запускает скрипт: sudo journalctl -u cron -f
  • Проверьте переменные: cron имеет минимальный PATH — указывайте полные пути к командам
🛟 Совет: Если скрипт работает в терминале, но не в cron — проблема почти всегда в окружении. В cron нет PATH, нет переменных. Используйте полные пути: /usr/bin/curl вместо curl, /bin/df вместо df.

✅ Задания

Базовые задания

  • Посмотри load average и сравни с количеством ядер (uptime + nproc)
  • Посмотри доступную память: free -h. Какая колонка — free или available — важнее?
  • Проверь использование диска и inodes: df -h и df -i
  • Посмотри сообщения ядра: dmesg -T | tail -30
  • Запусти vmstat 1 на 10 секунд, найди колонку id (idle CPU)
  • Напиши скрипт, который проверяет, запущен ли nginx, и выводит предупреждение через wall
  • Настрой отправку алерта в Telegram при нехватке места (> 80%)
  • Добавь скрипт проверки в cron на каждые 5 минут

🧩 Задачи со звёздочкой

  • Найдите процесс, который использует больше всего открытых файлов: проверьте /proc/PID/fd для топ-5 процессов по памяти
  • Настройте Prometheus + Node Exporter через Docker, откройте http://localhost:9090 и выполните запрос node_cpu_seconds_total
  • Создайте cron-задачу, которая каждый день в 6 утра отправляет в Telegram отчёт: диск, память, uptime, количество запущенных сервисов
