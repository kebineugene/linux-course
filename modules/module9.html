🌍 Сеть — кровеносная система сервера

Зачем админу сеть?

Без сети сервер бесполезен. Админ должен уметь:

  • 🔍 Диагностировать — почему не пингуется? почему DNS не резолвит?
  • 🚪 Управлять портами — кто слушает? что заблокировано?
  • 🔐 Туннелировать — как безопасно добраться до внутренней базы данных?
  • 🛡️ Защищать — фаервол, фильтрация трафика
💡 Аналогия: Сеть — это почтовая система. IP-адрес — адрес дома, порт — номер квартиры, DNS — телефонный справочник, фаервол — охрана на входе.

📖 Теория: IP-адреса и интерфейсы

Сетевые интерфейсы

Каждое сетевое подключение — это интерфейс. У сервера может быть несколько: физическая карта, виртуальная, loopback.

ip a # все интерфейсы и их IP-адреса ip a show eth0 # конкретный интерфейс ip r # таблица маршрутизации (куда идут пакеты) ip link show # статус интерфейсов (UP/DOWN)
🧠 Типичные интерфейсы:
  • lo — loopback (127.0.0.1), связь с самим собой
  • eth0 / ens33 — основной сетевой адаптер
  • docker0 — виртуальный мост Docker
  • wlan0 — Wi-Fi адаптер

Проверка доступности

ping -c 4 8.8.8.8 # 4 пинга до Google DNS ping -c 4 ya.ru # пинг по имени (проверяет и DNS) traceroute ya.ru # маршрут до хоста (какие роутеры на пути) mtr ya.ru # улучшенный traceroute (непрерывный)
💡 Пинг проверяет три вещи:
  1. Работает ли сетевой интерфейс
  2. Есть ли маршрут до цели
  3. Отвечает ли удалённый хост
Если ping 8.8.8.8 работает, а ping ya.ru нет — проблема в DNS, не в сети.

🔬 Практика: базовая диагностика

Пошаговая практика

ip a # мой IP-адрес ip r # шлюз по умолчанию ping -c 4 1.1.1.1 # есть ли интернет? ping -c 4 google.com # работает ли DNS? traceroute google.com # маршрут до Google

📖 DNS: как имена становятся адресами

Что такое DNS?

DNS (Domain Name System) — система, которая превращает google.com в 142.250.74.46. Без DNS пришлось бы запоминать IP-адреса.

Вы вводите: google.com
       │
       ▼
┌─────────────────┐
│  /etc/hosts      │ ← Проверяется ПЕРВЫМ
│  (локальная табл)│
└───────┬─────────┘
        │ не найдено
        ▼
┌─────────────────┐
│ /etc/resolv.conf │ ← Какой DNS-сервер спрашивать
│ nameserver 8.8.8.8│
└───────┬─────────┘
        │
        ▼
┌─────────────────┐
│ DNS-сервер       │ ← Отвечает: 142.250.74.46
│ (8.8.8.8)       │
└─────────────────┘

Команды для работы с DNS

dig google.com # полный DNS-запрос dig google.com +short # только IP-адрес dig MX google.com # почтовые записи dig @8.8.8.8 google.com # запрос к конкретному DNS-серверу nslookup google.com # простая альтернатива dig host google.com # ещё проще cat /etc/resolv.conf # какой DNS-сервер настроен cat /etc/hosts # локальная таблица имён
💡 /etc/hosts — быстрый способ: Можно вручную привязать имя к IP:
192.168.1.100 myserver.local
Полезно для локальной разработки и тестирования.

📖 Порты и сокеты

Что такое порт?

IP-адрес определяет компьютер, порт — программу на нём. На одном сервере работают десятки сервисов, и каждый слушает свой порт.

Сервер 192.168.1.10
├── :22   → SSH (sshd)
├── :80   → HTTP (nginx)
├── :443  → HTTPS (nginx)
├── :3306 → MySQL
└── :5432 → PostgreSQL
ss -tulpn # все слушающие порты и процессы # Расшифровка флагов: # t = TCP, u = UDP, l = listening, p = process, n = номера (не имена) ss -tulpn | grep :80 # кто слушает порт 80? ss -tnp # все TCP-соединения (установленные) ss -s # сводка по соединениям
🧠 Стандартные порты:
  • 22 — SSH
  • 80 — HTTP
  • 443 — HTTPS
  • 3306 — MySQL
  • 5432 — PostgreSQL
  • 6379 — Redis
  • 8080 — альтернативный HTTP

🛠️ cURL: универсальный сетевой инструмент

Что умеет cURL?

curl отправляет HTTP-запросы, скачивает файлы, проверяет заголовки. Незаменим для отладки API и веб-серверов.

curl http://example.com # получить страницу curl -I http://example.com # только заголовки (какой статус? 200? 404?) curl -O https://example.com/file.zip # скачать файл curl -s https://api.github.com # тихий режим (без прогресс-бара) curl -X POST -d '{"key":"val"}' \ -H "Content-Type: application/json" \ http://localhost:8080/api # отправить POST с JSON curl -o /dev/null -s -w "%{http_code}" http://localhost # только HTTP-код curl -k https://localhost # игнорировать самоподписанный SSL-сертификат
💡 Для скриптов: curl -o /dev/null -s -w "%{http_code}" вернёт только HTTP-код (200, 404, 500). Идеально для мониторинга.

🔐 SSH-туннели

Зачем туннели?

SSH-туннель — это зашифрованный канал через SSH-соединение. Позволяет безопасно обращаться к сервисам, которые не должны быть доступны из интернета.

ВАША МАШИНА                    СЕРВЕР
┌───────────┐    SSH Tunnel    ┌──────────────┐
│ localhost  │═════════════════│ 127.0.0.1     │
│ :8080      │    (encrypted)  │ :3306 (MySQL) │
└───────────┘                  └──────────────┘

Теперь MySQL доступна на вашем localhost:8080

Local Port Forwarding (самый частый)

Пробрасывает локальный порт на удалённый сервер:

ssh -L 8080:localhost:80 user@server # Теперь localhost:8080 на вашей машине → порт 80 на сервере ssh -L 3307:127.0.0.1:3306 user@db-server # Теперь localhost:3307 → MySQL на сервере # Флаги: ssh -L 8080:localhost:80 -N -f user@server # -N = не запускать оболочку (только туннель) # -f = работать в фоне

Remote Port Forwarding

Обратно: делает локальный сервис доступным на удалённом сервере:

ssh -R 9090:localhost:3000 user@server # Теперь на server:9090 доступен ваш localhost:3000 # Полезно для демонстрации локального сайта клиенту

Dynamic Forwarding (SOCKS-прокси)

ssh -D 1080 user@server # Создаёт SOCKS-прокси на localhost:1080 # Настройте браузер на использование SOCKS-прокси 127.0.0.1:1080 # Весь трафик браузера пойдёт через сервер
💡 Реальный кейс: Админ хочет подключиться к MySQL на production-сервере, но порт 3306 закрыт фаерволом. Решение: ssh -L 3307:127.0.0.1:3306 user@prod-server, затем подключиться через mysql -h 127.0.0.1 -P 3307.

🛡️ Фаервол и tcpdump

iptables (кратко)

sudo iptables -L -n -v # список всех правил sudo iptables -A INPUT -p tcp --dport 80 -j ACCEPT # разрешить HTTP sudo iptables -A INPUT -p tcp --dport 443 -j ACCEPT # разрешить HTTPS sudo iptables -D INPUT -p tcp --dport 80 -j ACCEPT # удалить правило # Для простого управления используйте UFW: sudo ufw allow 80/tcp sudo ufw allow 443/tcp sudo ufw status verbose

tcpdump: перехват пакетов

tcpdump позволяет видеть весь сетевой трафик в реальном времени. Мощный инструмент для глубокой диагностики.

sudo tcpdump -i any port 80 # весь HTTP-трафик sudo tcpdump -i eth0 host 8.8.8.8 # трафик с конкретным хостом sudo tcpdump -i any -w capture.pcap # записать в файл (открыть в Wireshark) sudo tcpdump -i any -c 100 # первые 100 пакетов sudo tcpdump -i any port 22 -n # SSH-трафик (без резолвинга имён)
💡 Совет: Для визуального анализа сохраните в .pcap и откройте в Wireshark (GUI-программа, есть на Windows/Mac/Linux).

🎯 Реальные сценарии

Сценарий 1: «Сайт не открывается» — диагностика за 5 шагов

# Шаг 1: Есть ли интернет? ping -c 4 8.8.8.8 # Шаг 2: Работает ли DNS? dig google.com +short # Шаг 3: Слушает ли веб-сервер? ss -tulpn | grep :80 # Шаг 4: Отвечает ли веб-сервер? curl -I http://localhost # Шаг 5: Что в логах? sudo journalctl -u nginx --since "5 min ago"

Сценарий 2: Подключение к удалённой БД через SSH-туннель

# 1. Открываем туннель (в отдельном терминале): ssh -L 5433:127.0.0.1:5432 -N -f user@db-server # 2. Подключаемся к БД через локальный порт: psql -h 127.0.0.1 -p 5433 -U postgres mydb # 3. Когда закончили — закрываем туннель: pkill -f "ssh -L 5433"

Сценарий 3: «Кто обращается к моему серверу?»

# Посмотреть активные соединения: ss -tnp # Посмотреть, кто слушает: ss -tulpn # Подсчитать соединения по IP: ss -tn | awk '{print $5}' | cut -d: -f1 | sort | uniq -c | sort -rn | head -10 # Перехватить трафик на порту 80: sudo tcpdump -i any port 80 -c 50

🔧 Troubleshooting: Если не получилось

❓ "Network is unreachable"
  • Нет маршрута по умолчанию — проверьте: ip r
  • Интерфейс выключен: sudo ip link set eth0 up
  • Проверьте кабель / виртуальную сеть
❓ "Could not resolve host" / DNS не работает
  • Проверьте DNS-сервер: cat /etc/resolv.conf
  • Попробуйте пинг по IP: ping 8.8.8.8
  • Если IP работает, а имя нет — проблема точно в DNS
  • Попробуйте: dig @8.8.8.8 google.com
❓ "Connection refused"
  • Сервис не запущен или не слушает нужный порт
  • Проверьте: sudo ss -tulpn | grep :PORT
  • Запустите сервис: sudo systemctl start nginx
❓ "Connection timed out"
  • Фаервол блокирует — проверьте: sudo ufw status или sudo iptables -L
  • Сервер не отвечает — проверьте: ping server_ip
  • Неправильный порт — проверьте, какой порт слушает сервис
❓ SSH-туннель не работает
  • Проверьте, что SSH-соединение установлено
  • Убедитесь, что целевой сервис запущен на сервере
  • Попробуйте с -v: ssh -v -L 8080:localhost:80 user@server
  • Проверьте: ss -tulpn | grep 8080 на локальной машине
🛟 Универсальный алгоритм:
1. ping 8.8.8.8 — есть интернет?
2. ping ya.ru — работает DNS?
3. ss -tulpn — сервис слушает?
4. curl localhost:PORT — сервис отвечает?
5. sudo ufw status — фаервол не блокирует?

✅ Задания

Базовые задания

  • Проверь свой IP-адрес: ip a
  • Проверь таблицу маршрутизации: ip r
  • Пингни 4 раза 1.1.1.1: ping -c 4 1.1.1.1
  • Проверь DNS: dig google.com +short
  • Посмотри, какие порты слушает твой сервер: ss -tulpn
  • Отправь GET-запрос к любому сайту через curl, посмотри заголовки: curl -I https://ya.ru
  • Проверь маршрут до google.com: traceroute google.com
  • Посмотри содержимое /etc/resolv.conf и /etc/hosts
  • Запусти tcpdump на порт 80 и в другом терминале сделай curl localhost
  • Добавь запись в /etc/hosts: 127.0.0.1 myapp.local и проверь ping myapp.local

🧩 Задачи со звёздочкой

  • Создай SSH-туннель к своему серверу: пробрось порт 80 на локальный 8080, проверь через curl localhost:8080
  • Напиши скрипт, который проверяет доступность 5 сайтов (через curl -o /dev/null -s -w "%{http_code}") и выводит статус каждого
  • Найди топ-10 IP-адресов из access.log nginx, используя ss или awk
  • Захвати 100 пакетов через tcpdump в файл, затем посмотри через tcpdump -r
