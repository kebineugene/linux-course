🛒 Установка ПО

Зачем админу пакеты и сервисы?

Пакеты — это способ распространения программ. Сервисы — программы, работающие в фоне. Админ должен уметь:

  • Устанавливать и обновлять ПО — безопасно, с проверкой зависимостей
  • ⚙️ Управлять сервисами — запускать, останавливать, настраивать автозапуск
  • 🔍 Находить нужные пакеты — когда не знаешь точное название
  • 🛡️ Обновлять систему — закрывать уязвимости безопасности
💡 Аналогия: Пакетный менеджер — это как App Store, но для сервера. Ты говоришь "хочу nginx", и система сама скачивает программу, все её зависимости, устанавливает и настраивает.

📖 Теория: Пакетные менеджеры

Debian/Ubuntu: apt

apt (Advanced Package Tool) — основной пакетный менеджер в Debian/Ubuntu. Работает с репозиториями — серверами, где хранятся пакеты.

sudo apt update # обновить список пакетов из репозиториев sudo apt upgrade # обновить все установленные пакеты sudo apt install nginx # установить nginx sudo apt remove nginx # удалить (конфиги останутся) sudo apt purge nginx # удалить вместе с конфигами sudo apt autoremove # удалить ненужные зависимости
Разница между remove и purge:
  • remove — удаляет программу, но оставляет конфиги в /etc/
  • purge — удаляет всё, включая конфиги (чистый снос)

Поиск и информация о пакетах

apt search "web server" # поиск по описанию apt show nginx # подробная информация о пакете dpkg -l | grep nginx # проверить, установлен ли пакет apt list --installed # список всех установленных пакетов apt-cache policy nginx # показать версии и репозитории

Зависимости пакетов

Пакеты часто зависят друг от друга. Например, nginx зависит от библиотек OpenSSL. apt автоматически устанавливает все зависимости.

apt depends nginx # показать зависимости пакета apt rdepends nginx # показать, кто зависит от nginx sudo apt install -f # исправить сломанные зависимости
⚠️ Никогда не удаляйте пакеты вручную через rm — это сломает базу данных пакетов. Используйте только apt remove или dpkg --remove.

Обновление системы безопасности

Регулярное обновление критично для безопасности сервера:

sudo apt update && sudo apt upgrade -y # Или только обновления безопасности: sudo apt upgrade --only-upgrade
💡 Совет: На production-серверах настройте автоматические обновления безопасности через пакет unattended-upgrades.

🔬 Практика: apt

Пошаговая практика

sudo apt update apt search htop sudo apt install -y htop dpkg -l | grep htop apt show htop sudo apt remove htop sudo apt purge htop sudo apt autoremove

Обновление системы

sudo apt update sudo apt upgrade -y # Проверить, что нужно обновить: apt list --upgradable
⚠️ Внимание: Перед обновлением production-сервера всегда делайте бэкап! Некоторые обновления могут сломать совместимость.

📚 Репозитории

Что такое репозитории?

Репозитории — это серверы, где хранятся пакеты. Список репозиториев находится в /etc/apt/sources.list и файлах /etc/apt/sources.list.d/.

cat /etc/apt/sources.list ls /etc/apt/sources.list.d/

Добавление PPA (Personal Package Archive)

PPA — дополнительные репозитории от сообщества. Позволяют установить более новые версии программ.

sudo add-apt-repository ppa:nginx/stable sudo apt update sudo apt install nginx
⚠️ Осторожно с PPA: Добавляйте только проверенные репозитории. Неофициальные PPA могут содержать вредоносное ПО.

Поиск пакета по файлу

Если вы знаете имя файла, но не знаете, в каком пакете он находится:

apt-file search nginx.conf # Или через dpkg: dpkg -S nginx.conf

📦 Другие способы установки ПО

Snap-пакеты

Snap — универсальный пакетный формат от Canonical. Пакеты изолированы, обновляются автоматически, работают на любом дистрибутиве.

sudo snap install vlc snap list sudo snap remove vlc sudo snap refresh # обновить все snap-пакеты
💡 Плюсы Snap: Изоляция, автообновление, кроссплатформенность.
Минусы: Больший размер, медленнее запуск, меньше контроля.

Flatpak (альтернатива Snap)

Flatpak — ещё один универсальный формат, популярный в GNOME-окружении.

flatpak install flathub org.gimp.GIMP flatpak list flatpak uninstall org.gimp.GIMP

AppImage

AppImage — переносимый исполняемый файл. Не требует установки, работает из коробки.

chmod +x someapp.AppImage ./someapp.AppImage
💡 Когда использовать AppImage: Когда нужно быстро протестировать программу без установки, или когда нужна конкретная версия.

Компиляция из исходников (базово)

Иногда нужно собрать программу из исходного кода. Это даёт максимальный контроль, но сложнее.

./configure # проверка зависимостей make # компиляция sudo make install # установка
⚠️ Внимание: Программы, установленные через make install, не управляются пакетным менеджером. Их сложно обновлять и удалять.

Теория: Сервисы

Что такое сервис?

Сервис (демон) — программа, работающая в фоне без интерактивного ввода. Например: веб-сервер nginx, база данных MySQL, SSH-сервер.

Управление сервисами через systemctl

sudo systemctl status nginx # статус сервиса sudo systemctl start nginx # запустить sudo systemctl stop nginx # остановить sudo systemctl restart nginx # перезапустить sudo systemctl reload nginx # перечитать конфиги без остановки sudo systemctl enable nginx # автозапуск при загрузке sudo systemctl disable nginx # отключить автозапуск sudo systemctl mask nginx # полностью заблокировать запуск
🧠 Разница между restart и reload:
  • restart — полностью остановить и запустить заново (короткий downtime)
  • reload — перечитать конфиги без остановки (zero downtime)

Просмотр логов сервиса

Systemd собирает логи всех сервисов. Используйте journalctl для просмотра:

sudo journalctl -u nginx # логи nginx sudo journalctl -u nginx -f # следить за логами в реальном времени sudo journalctl -u nginx --since today # логи за сегодня sudo journalctl -u nginx -n 50 # последние 50 строк
💡 Критично для админа: Если сервис не запускается — первым делом смотрите логи через journalctl -u <сервис>.

Список сервисов

systemctl list-units --type=service # все активные сервисы systemctl list-unit-files --type=service # все unit-файлы systemctl --failed # упавшие сервисы

🔬 Практика: systemctl

Пошаговая практика

sudo systemctl list-units --type=service sudo systemctl start ssh sudo systemctl status ssh sudo systemctl enable ssh sudo journalctl -u ssh -n 20 sudo systemctl restart ssh sudo systemctl stop ssh

Работа с nginx

sudo apt install -y nginx sudo systemctl start nginx sudo systemctl enable nginx sudo systemctl status nginx curl localhost # проверить, что nginx отвечает sudo journalctl -u nginx -f # следить за логами (Ctrl+C для выхода)

🎯 Реальные сценарии

Сценарий 1: Установка веб-сервера с нуля

# Обновить систему sudo apt update && sudo apt upgrade -y # Установить nginx sudo apt install -y nginx # Запустить и добавить в автозагрузку sudo systemctl start nginx sudo systemctl enable nginx # Проверить статус sudo systemctl status nginx # Открыть браузер и перейти на http://

Сценарий 2: Обновление системы безопасности

# Проверить доступные обновления sudo apt update apt list --upgradable # Установить только обновления безопасности sudo apt upgrade -y # Перезагрузить, если обновилось ядро sudo reboot
Автоматизация: На production-серверах настройте unattended-upgrades для автоматической установки обновлений безопасности.

Сценарий 3: Сервис не запускается — диагностика

# 1. Проверить статус sudo systemctl status nginx # 2. Посмотреть логи sudo journalctl -u nginx -n 50 # 3. Проверить конфиг sudo nginx -t # 4. Исправить ошибку в конфиге sudo nano /etc/nginx/nginx.conf # 5. Перезапустить sudo systemctl restart nginx

Сценарий 4: Найти, какой пакет содержит нужный файл

# 1. Установить apt-file (если не установлен) sudo apt install -y apt-file sudo apt-file update # 2. Найти пакет apt-file search nginx.conf # 3. Посмотреть содержимое пакета apt-file list nginx

🔧 Troubleshooting: Если не получилось

❓ "E: Unable to locate package"
  • Пакет не найден в репозиториях
  • Выполните sudo apt update — возможно, списки устарели
  • Проверьте название: apt search <часть_имени>
  • Возможно, нужен дополнительный репозиторий (PPA)
❓ "Unmet dependencies" / "Broken packages"
  • Конфликт зависимостей
  • Выполните: sudo apt install -f — исправить сломанные зависимости
  • Или: sudo apt --fix-broken install
  • В крайнем случае: sudo dpkg --configure -a
❓ "Failed to start <service>.service"
  • Сервис не запустился
  • Смотрите логи: sudo journalctl -u <сервис> -n 50
  • Проверьте конфиг: sudo <сервис> -t (например, nginx -t)
  • Проверьте права на файлы и порты
❓ "No space left on device" при установке
  • Не хватает места на диске
  • Проверьте: df -h
  • Очистите кэш: sudo apt clean
  • Удалите старые ядра: sudo apt autoremove
❓ "The following packages have been kept back"
  • Некоторые пакеты не обновляются автоматически
  • Выполните: sudo apt dist-upgrade (осторожно!)
  • Или обновите конкретно: sudo apt install <пакет>
🛟 Универсальный совет: Если что-то сломалось после установки пакета — посмотрите /var/log/apt/history.log, чтобы понять, что именно установилось.

✅ Задания

Базовые задания

  • Обнови списки пакетов: sudo apt update
  • Найди пакет htop: apt search htop
  • Установи htop: sudo apt install -y htop
  • Проверь, что htop установлен: dpkg -l | grep htop
  • Удали htop полностью: sudo apt purge htop
  • Установи nginx и запусти сервис
  • Добавь nginx в автозагрузку: sudo systemctl enable nginx
  • Проверь статус nginx: sudo systemctl status nginx
  • Посмотри логи nginx: sudo journalctl -u nginx -n 20
  • Перезапусти nginx: sudo systemctl restart nginx
  • Установи любой snap-пакет (например, snap install curl)
  • Скачай AppImage любой программы и запусти её

🧩 Задачи со звёздочкой

  • Найдите, в каком пакете находится файл nginx.conf: apt-file search nginx.conf
  • Покажите все зависимости nginx: apt depends nginx
  • Добавьте PPA репозиторий и установите пакет из него
  • Настройте автоматические обновления безопасности через unattended-upgrades
  • Создайте свой systemd-сервис, который запускает скрипт каждые 5 минут (используйте timer)
  • Найдите все упавшие сервисы: systemctl --failed и попробуйте их исправить
