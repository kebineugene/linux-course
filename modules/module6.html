🛡️ Безопасность — это привычка

Зачем админу безопасность?

Безопасность в Linux строится на принципе наименьших привилегий: пользователь или сервис должен иметь ровно столько прав, сколько нужно для работы, и ни битом больше.

  • 👥 Управление доступом — кто может входить в систему и что делать
  • 🔑 Аутентификация — как подтверждать личность (ключи вместо паролей)
  • 🚪 Защита периметра — фаерволы и защита от брутфорса
  • 📝 Аудит — понимание, кто и что делал в системе
💡 Золотое правило: Никогда не работайте под root в повседневной жизни. Используйте обычного пользователя и sudo для привилегированных команд.

Теория: Пользователи и группы

Где хранятся пользователи?

В Linux вся информация о пользователях хранится в текстовых файлах. Это удобно для администрирования и бэкапов.

cat /etc/passwd # список всех пользователей (имя, UID, домашняя папка, оболочка) cat /etc/shadow # хеши паролей (доступен только root!) cat /etc/group # список групп
🧠 Формат /etc/passwd: username:x:UID:GID:comment:home_dir:shell
Если оболочка /usr/sbin/nologin или /bin/false — пользователь не может войти в систему (используется для сервисов).

Управление пользователями

sudo useradd -m -s /bin/bash username # создать с домашней папкой и bash sudo passwd username # установить пароль sudo userdel -r username # удалить пользователя и его домашнюю папку sudo usermod -aG groupname username # ДОБАВИТЬ в группу (ВАЖНО: флаг -a!)
⚠️ Критично: Команда usermod -G group user (без флага -a) удаляет пользователя из всех остальных групп! Всегда используйте -aG (append).

🔬 Практика: пользователи

Пошаговая практика

sudo useradd -m -s /bin/bash developer sudo passwd developer sudo groupadd webdevs sudo usermod -aG webdevs developer id developer # проверить UID и группы su - developer # переключиться на пользователя groups # проверить текущие группы exit # вернуться назад

Теория: Sudo и привилегии

Как работает sudo?

sudo позволяет запускать команды от имени другого пользователя (обычно root). Настройки хранятся в /etc/sudoers.

sudo command # выполнить команду от root sudo -u username cmd # выполнить от имени конкретного пользователя sudo -l # посмотреть свои sudo-права sudo -i # получить root-оболочку (аналог su -)
💡 Совет: Команда sudo -l — первая, которую должен выполнить админ, получив доступ к новому серверу, чтобы понять свои полномочия.

Редактирование sudoers

Никогда не редактируйте /etc/sudoers обычным редактором! Используйте visudo — он проверяет синтаксис перед сохранением. Ошибка в этом файле может заблокировать доступ к sudo навсегда.

sudo visudo # Добавить полные права пользователю: username ALL=(ALL:ALL) ALL # Разрешить только конкретную команду без пароля: username ALL=(ALL) NOPASSWD: /usr/bin/systemctl restart nginx # Разрешить всю группу: %adminbot ALL=(ALL) ALL
🚨 Опасно: Если вы сохраните файл с синтаксической ошибкой через обычный редактор — вы потеряете sudo. visudo не даст этого сделать.

🔬 Практика: sudoers

Настройка прав

sudo -l # посмотреть свои права sudo visudo # открыть редактор # Добавить: developer ALL=(ALL) NOPASSWD: /usr/bin/apt update sudo -u developer sudo apt update # проверить, что работает без пароля

SSH: ключи и защита

Аутентификация по ключам

Пароли можно подобрать. Криптографические ключи (ED25519 или RSA 4096) — практически нет. Это стандарт для любых серверов.

# Генерация ключа (на локальной машине): ssh-keygen -t ed25519 -C "admin@company.com" # Копирование публичного ключа на сервер: ssh-copy-id user@server_ip # Ручное копирование (если ssh-copy-id нет): cat ~/.ssh/id_ed25519.pub | ssh user@server "mkdir -p ~/.ssh && cat >> ~/.ssh/authorized_keys && chmod 700 ~/.ssh && chmod 600 ~/.ssh/authorized_keys"
🧠 Права на SSH-файлы критичны!
  • ~/.ssh → 700 (drwx------)
  • ~/.ssh/authorized_keys → 600 (-rw-------)
  • Если права шире — SSH-сервер откажется принимать ключи!

Hardening SSH-сервера

Настройки находятся в /etc/ssh/sshd_config. После изменений нужен рестарт: sudo systemctl restart sshd

PermitRootLogin no # запретить вход под root PasswordAuthentication no # только ключи (сначала убедитесь, что ключ работает!) PubkeyAuthentication yes # разрешить ключи Port 2222 # сменить порт (защита от простых ботов, не панацея) MaxAuthTries 3 # максимум попыток ввода AllowUsers admin developer # разрешить вход только конкретным пользователям
🚨 КРИТИЧНО: Перед тем как отключить парольную аутентификацию или сменить порт, откройте второй терминал и проверьте, что вы можете войти по ключу. Иначе вы заблокируете себя снаружи!

Fail2ban и UFW

Fail2ban: защита от брутфорса

Fail2ban сканирует логи и банит IP-адреса, с которых происходит слишком много неудачных попыток входа. Обязателен для любого сервера, доступного из интернета.

sudo apt install -y fail2ban sudo systemctl enable --now fail2ban sudo fail2ban-client status sshd # посмотреть статус sudo fail2ban-client set sshd unbanip 1.2.3.4 # разбанить IP

UFW: простой фаервол

UFW (Uncomplicated Firewall) — надстройка над iptables. Прост в использовании.

sudo ufw allow ssh # разрешить SSH (порт 22) sudo ufw allow 80/tcp # разрешить HTTP sudo ufw allow 443/tcp # разрешить HTTPS sudo ufw allow from 192.168.1.0/24 # разрешить только из локальной сети sudo ufw status verbose # посмотреть правила sudo ufw enable # включить фаервол sudo ufw disable # выключить
⚠️ Внимание: Всегда выполняйте sudo ufw allow ssh ДО включения фаервола (ufw enable). Иначе вы потеряете доступ к серверу!

🔧 Troubleshooting: Если не получилось

❓ "Permission denied (publickey)" при входе по SSH
  • Проверьте права на ~/.ssh (700) и authorized_keys (600) на сервере
  • Проверьте, что публичный ключ скопирован полностью (одной строкой)
  • Попробуйте войти с отладкой: ssh -v user@server
  • Проверьте PubkeyAuthentication yes в sshd_config
❓ "sudo: no valid sudoers sources found"
  • Пользователь не добавлен в группу sudo/wheel
  • Выполните от root: usermod -aG sudo username
  • Или проверьте синтаксис /etc/sudoers через visudo -c
❓ Заблокировал себя на сервере после настройки SSH/UFW
  • Используйте консоль восстановления через панель хостинга (VNC/Console)
  • Войдите как root, исправьте /etc/ssh/sshd_config или выполните ufw disable
  • Перезапустите SSH: systemctl restart sshd
❓ "Bad owner or permissions on /home/user/.ssh/config"
  • SSH очень строг к правам. Владелец файлов должен быть пользователем, а не root
  • Исправление: chown -R $USER:$USER ~/.ssh && chmod 700 ~/.ssh && chmod 600 ~/.ssh/*
🛟 Универсальный совет: Если что-то пошло не так с SSH — смотрите логи: sudo journalctl -u ssh -f в реальном времени, пока пытаетесь подключиться. Там будет точная причина отказа.
